# ADR-0002: 이미지 벡터화 전략 — 비전 모델 캡셔닝 후 텍스트 임베딩

- **상태**: Accepted
- **결정일**: 2026-06-30
- **결정자**: 개발팀
- **관련 ADR**: ADR-0001
- **영향 받는 코드**: `core/src/.../service/ImageCaptioningService.java`, `app-widget/src/.../service/KnowledgeIngestionService.java`

## 컨텍스트 (Why)

docx·pdf 등 문서에 포함된 이미지(그래프·다이어그램·스캔 이미지)를 RAG 검색에 활용하려면
이미지를 어떤 방식으로 벡터화할지 결정해야 한다.

현재 임베딩 모델은 `bge-m3`(텍스트 전용)이며, 이미지를 직접 받는 멀티모달 임베딩 모델은
미도입 상태다.

## 결정 (What)

```
이미지를 직접 임베딩하지 않는다.
대신 Ollama 비전 모델(예: qwen2.5vl)로 이미지 캡션 텍스트를 생성한 뒤,
해당 텍스트를 마크다운 본문에 인라인 삽입한다.
최종 마크다운이 기존 텍스트 청킹·bge-m3 임베딩 파이프라인을 그대로 거친다.

비전 모델 미설정(WIDGET_VISION_MODEL 환경변수 미지정) 또는 캡셔닝 실패 시
이미지는 조용히 건너뛰고(경고 로그만), 텍스트 인입을 차단하지 않는다.
```

## 결과 (Consequences)

### 장점
- **파이프라인 무변경**: 이미지도 결국 텍스트로 변환되므로 임베딩·청킹·검색 코드 수정 불필요.
- **멀티모달 임베딩 모델 불필요**: 인프라 복잡도 증가 없음.
- **선택적 활성화**: 비전 모델 없이도 텍스트 파싱은 정상 동작. 점진적 도입 가능.
- **검색 가능성**: 캡션 텍스트가 청크에 포함되어 이미지 내용으로 검색 가능.

### 단점·트레이드오프
- **캡션 품질 의존**: 캡션 정확도가 비전 모델 능력에 달려 있음. 그래프·수식은 캡션만으로 의미 손실 가능.
- **추가 레이턴시**: 이미지당 Ollama 비전 모델 호출 → 대량 이미지 문서 인입 시간 증가.
- **비전 모델 추가 운영**: Ollama에 별도 비전 모델 pull 필요(`OLLAMA_MAX_LOADED_MODELS=2` 이상 설정).

### 후속 작업
- 비전 모델 권장 사양 문서화 (qwen2.5vl:3b vs 7b 성능 비교)
- 이미지 수 상한(MAX_IMAGES=50) 실운영 검증

## 대안 (검토했으나 채택 안 한 옵션)

### 옵션 A — 멀티모달 임베딩 모델 도입 (clip, llava-embedding 등)
- 이미지를 벡터 공간에 직접 임베딩. 텍스트-이미지 교차 검색 가능.
- **채택 안 한 이유**: 기존 bge-m3 텍스트 임베딩과 벡터 공간이 달라 검색 파이프라인 전면 수정 필요. 도입 비용 대비 이점 불명확.

### 옵션 B — 이미지 무시 (텍스트만 추출)
- 구현 가장 단순.
- **채택 안 한 이유**: 이미지에 핵심 정보(차트·표 이미지·스캔본)가 담긴 경우 RAG 품질 저하.

### 옵션 C — 이미지를 별도 오브젝트 스토리지에 저장 후 링크만 청크에 포함
- 원본 이미지 보존 가능.
- **채택 안 한 이유**: 오브젝트 스토리지 인프라 추가 필요. 링크만으로는 검색에 활용 불가.

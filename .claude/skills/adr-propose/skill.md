# ADR 제안 스킬

대화·계획·코드 작성 중 **결정(decision)** 을 감지하여 `docs/adr/NNNN-slug.md` 작성을 사용자에게 제안한다.

## 언제 트리거하는가

다음 패턴 중 하나가 보이면 즉시 이 스킬을 발동한다:

### 명시적 결정 시그널
- 사용자: "옵션 X 로 가자", "이거 결정하자", "이 방향으로 진행해", "이렇게 가자", "A 로 해줘"
- 사용자: "확정", "최종", "정하자"
- 어시스턴트가 옵션 A/B/C/D 를 제안하고 사용자가 그 중 하나를 선택

### 암묵적 결정 시그널
- 두 문서·두 코드 사이 모순을 해결하기로 합의
- 새 라이브러리·프레임워크·서비스 채택 (예: "Tika 도입", "SES 사용")
- 비용·일정·SLA·임계값 등의 숫자를 확정
- 운영 정책 결정 (예: 파일 크기 30MB, 타임아웃 600초)
- "Phase 0 에 도입", "Phase 1+ 로 미룸" 결정

### 트리거하지 않는 경우
- 단순 정보 전달 ("이건 X 입니다")
- 결정 후보를 검토 중이지만 아직 미결정
- ADR 이 이미 있는 결정의 재인용

## 워크플로우

### 1. 결정 감지 시 즉시 사용자에게 제안

```
어시스턴트 메시지 끝부분에 아래 형태로 제안:

---
📋 **ADR 후보 감지**

위 결정사항은 ADR-NNNN 로 기록할 가치가 있어 보입니다.

제목 후보: "{한 줄 제목}"
영향 받는 문서: {requirements/X.md, ...}

ADR 로 남길까요? (y/n)
```

사용자가 y 하면 step 2 로. n 하면 스킵.

### 2. ADR 번호 결정

```
docs/adr/ 디렉토리를 ls 또는 grep 으로 확인:
- 가장 큰 번호 + 1 = 새 ADR 번호
- 4자리 zero-pad (예: 0006)
```

### 3. ADR 작성

`docs/adr/0000-template.md` 형식을 따라 새 파일 작성. 필수 섹션:

```yaml
ADR-NNNN: 결정 제목

- 상태: Accepted
- 결정일: {오늘 날짜}
- 결정자: {사용자 이름 또는 "사용자 + 시니어 백엔드 엔지니어"}
- 관련 ADR: {이전 ADR 번호들}
- 영향 받는 문서: requirements/XX 섹션 N, ...

## 컨텍스트 (Why)
- 무엇이 문제였고 왜 결정이 필요했는지
- 대화에서 추출 — 시간 흘러도 이해 가능하게 자체 완결적으로

## 결정 (What)
- 무엇을 결정했는지 단정형
- 코드 영향 한 줄로

## 결과 (Consequences)
### 장점
- 대화에서 사용자 + 어시스턴트가 짚은 장점 정리

### 단점·트레이드오프
- 대화에서 짚은 단점 정리. "trade-off 없음" 같은 거짓말 금지.

### 후속 작업
- 어느 코드·문서·인프라가 영향 받는지

## 대안 (검토했으나 채택 안 한 옵션)
- 어시스턴트가 제시한 옵션 A/B/C/D 중 채택 안 한 것
- 각 옵션의 채택 안 한 이유 (대화에서 추출)

## 참고
- requirements/XX 권위 출처
- 관련 ADR
```

### 4. README.md 인덱스 갱신

`docs/adr/README.md` 의 ADR 목록 표에 새 항목 추가:

```markdown
| [NNNN](NNNN-slug.md) | 제목 | Accepted | YYYY-MM-DD |
```

### 5. 영향 받는 requirements 문서 업데이트 안내

ADR 작성 후 어시스턴트가 사용자에게 확인:
"requirements/XX.md 의 관련 섹션도 갱신할까요? (이 ADR 을 권위 출처로 인용)"

## 작성 원칙

### 자체 완결성 (Self-contained)
ADR 은 작성 시점의 대화 맥락 없이도 6개월 후에 읽어 이해 가능해야 한다.
- "위에서 논의한 대로" 같은 표현 금지
- 대화 외부 참조는 항상 권위 출처(requirements/, 다른 ADR) 명시

### 단정형 (Imperative tone)
"~할 수 있다"가 아니라 "~한다", "~를 채택한다", "~를 거부한다".

### 대안의 정직한 기록
검토했으나 안 한 옵션을 **장점·단점 모두** 기록. ADR 의 가치는 미래의 누군가가 "왜 옵션 B 가 아닌 A 인가?" 물을 때 답을 주는 것.

### 단점 숨기지 마라
모든 결정에는 trade-off 가 있다. "단점 없음"이라 쓰지 마라. 단점이 떠오르지 않으면 결정을 다시 살펴라.

### 크기 가이드
- 한 ADR = 80~200줄
- 너무 짧으면 (50줄 미만) 컨텍스트·대안 부족
- 너무 길면 (300줄+) 두 ADR 로 분할 검토

## 트리거 검증 예시

### Should-trigger (이 스킬 발동)
- "옵션 A 로 해" — 어시스턴트가 옵션 제시 후 사용자 선택
- "Tavily 말고 Tavily 보류, SearXNG self-hosted 로 가자"
- "타임아웃 600초로 해"
- "비용은 문서 기준으로 통일"
- "이거 ADR 로 남기자"
- "Phase 1+ 로 미루자"

### Should-NOT-trigger
- "Spring Boot 가 뭐야?" (정보 조회)
- "코드 컴파일 됐어?" (상태 확인)
- "이거 어떻게 동작해?" (설명 요청)
- 이미 ADR-NNNN 에 결정된 사항을 재인용하는 경우
- 어시스턴트가 옵션 검토 중이고 아직 사용자 선택 전

## ADR backfill 모드

요청: "지금까지 결정사항 중 ADR 누락된 거 backfill 해줘"

1. 대화 히스토리 + `requirements/` + 기존 `docs/adr/` 비교
2. ADR 누락 후보 목록 작성
3. 사용자에게 우선순위 확인 (모두? 위험도 큰 것만?)
4. 위 워크플로우 반복 적용
